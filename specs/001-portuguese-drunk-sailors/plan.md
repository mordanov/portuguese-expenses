# Implementation Plan: Portuguese Drunk Sailors

**Branch**: `001-portuguese-drunk-sailors` | **Date**: 2026-05-27 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/001-portuguese-drunk-sailors/spec.md`

## Summary

Build "Portuguese Drunk Sailors" — a full-stack family expense tracking and cost allocation
web application. A family of 8 (6 adults, 2 children) shares purchases across stores; after
each trip one person pays the full bill. The app captures receipts via OCR (GPT-4o vision),
lets users review/edit extracted items, allocates each item to the family members who consumed
it, distributes ticket-level discounts proportionally, and computes pairwise net balances.

Implementation is delivered via parallel AI agents coordinated through `run-agents.sh`
(Brainstorm MCP). The coordinator (human) reviews all agent handoffs before acceptance.

## Technical Context

**Language/Version**: Python 3.12 (backend), TypeScript 5.x strict (frontend)
**Primary Dependencies**: FastAPI, SQLAlchemy 2.x async, asyncpg, Alembic, PyJWT (RS256),
  bcrypt, pdf2image, openai SDK v1.x (backend) | React 18, TanStack Query v5,
  React Hook Form, Zod, i18next, HeroUI, Tailwind CSS, Vitest, MSW (frontend)
**Storage**: PostgreSQL 16 — all monetary as NUMERIC(10,2), UUIDs for all PKs
**Testing**: pytest + pytest-asyncio + httpx + pytest-cov ≥80% (backend);
  Vitest + React Testing Library + MSW (frontend)
**Target Platform**: Linux containers via Docker Compose (db + backend + frontend)
**Project Type**: Web application (REST API backend + SPA frontend)
**Performance Goals**: Receipt OCR draft within 15s; ticket confirm within 3s;
  balance screen load within 2s for 500 tickets
**Constraints**: All monetary arithmetic uses Python `Decimal`; no floats for money;
  CORS locked to frontend origin; JWT RS256 only; OCR always mocked in tests
**Scale/Scope**: ~8 users, household scale; ~9 UI pages; ~19 API endpoints

## Agent Execution Notes

Implementation is delivered via `run-agents.sh` (Brainstorm MCP). Key guidelines:

1. **Tasks must be self-contained**: Each agent task must include the exact file path,
   the relevant contract section, and the specific entity or endpoint being implemented.
2. **Agents do not share memory**: All necessary context (schemas, entity rules, endpoint
   contracts) must be embedded in or linked from the task description.
3. **Coordinator review**: The human coordinator reviews every handoff before it is
   accepted. Agents send completed work as Brainstorm handoff messages.
4. **Layer parallelism**: Backend models, repositories, services, and routers for
   different domains can be implemented in parallel by different agents.
5. **Frontend/backend parallelism**: Once API contracts are finalized, frontend and
   backend agents can work in parallel.
6. **Constitution gates**: Every agent task implicitly carries the full constitution.
   Agents must not introduce floats for money, synchronous DB calls, hardcoded strings,
   or any other constitution violations.
